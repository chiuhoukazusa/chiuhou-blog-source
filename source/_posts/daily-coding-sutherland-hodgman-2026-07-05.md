## 背景与动机

在计算机图形学中，**多边形裁剪**（Polygon Clipping）是一个基础且重要的操作。无论你是在开发一个 2D UI 框架、一个地图渲染引擎，还是 3D 渲染管线中的三角形裁剪，都离不开多边形裁剪技术。

### 为什么需要多边形裁剪？

考虑以下场景：你有一个复杂的多边形（比如一个星星形状），但显示区域只限于一个矩形窗口。如果直接把整个多边形画上去，超出窗口的部分会被屏幕硬件自动裁剪——但你得到的只是一个被"砍断"的像素结果，而不是一个真正位于窗口内的裁剪后多边形。

实际应用中，裁剪后的多边形被用于：

- **视锥剔除（Frustum Culling）**：在 3D 渲染中，三角形被视锥体裁剪成位于视锥体内的子三角形。OpenGL 和 DirectX 的渲染管线都包含这个阶段。
- **GUI 裁剪**：窗口系统中的视图裁剪——当一个组件部分被父组件遮挡时，只需渲染可见部分。macOS 的 Quartz 合成器和 Windows 的 DWM 都依赖此技术。
- **地图渲染**：只渲染当前视口范围内的地理要素。Google Maps 和 Mapbox 在客户端做视口裁剪以减少绘制调用。
- **GPU 管线**：现代 GPU 的光栅化流水线中，裁剪是固定功能阶段之一，发生在顶点着色器之后、片元着色器之前。
- **CAD 软件**：AutoCAD 和 SolidWorks 使用多边形裁剪来实现视图裁剪和图形布尔运算。

### 为什么选择 Sutherland-Hodgman 算法？

多边形裁剪有几大经典算法：

| 算法 | 适用对象 | 时间复杂度 | 特点 |
|------|---------|-----------|------|
| Sutherland-Hodgman (1974) | 凸裁剪窗口 | O(n) | 逐边裁剪，简单优雅 |
| Weiler-Atherton (1977) | 任意形状窗口 | O(n²) | 可处理凹窗口，但复杂 |
| Vatti (1992) | 任意形状 | O((n+m)log(n+m)) | 扫描线，处理复杂情况 |
| Greiner-Hormann (1998) | 任意形状窗口 | O(n²) | 基于链表，实现复杂 |

**Sutherland-Hodgman 是入门多边形裁剪的完美起点**：

1. 算法极其简洁——只需处理 4 种顶点情况
2. 对于凸裁剪窗口（如矩形视口），效率为 O(n)，每次迭代只遍历多边形顶点一次
3. 它是理解更复杂裁剪算法（Weiler-Atherton、Vatti）的基础
4. 被广泛用于 GPU 管线的裁剪阶段，几乎所有图形 API 的内部实现都源于此

> 前两天我们实现了 Cohen-Sutherland 和 Liang-Barsky **直线裁剪**算法。今天的 Sutherland-Hodgman 是这"裁剪三部曲"的终章——从线段裁剪升级到多边形裁剪，完成从"线"到"面"的跃迁。如果说直线裁剪告诉你"哪些线段可见"，那么多边形裁剪告诉你"哪些区域可见"——这是从 2D 线框到 2D 填充的关键一步。

## 核心原理

### 核心思想：逐边裁剪（Edge-by-Edge Clipping）

Sutherland-Hodgman 的核心思想非常直观：**把裁剪问题分解为 4 次独立的半平面裁剪**。

对于一个矩形裁剪窗口，它有 4 条边：左、右、上、下。我们不需要一次性处理所有 4 条边——只需要按顺序一条一条地裁剪：

```
原始多边形 → 左边界裁剪 → 上边界裁剪 → 右边界裁剪 → 下边界裁剪 → 裁剪后多边形
```

每次裁剪只关注**当前这条边**：把"在边界内侧"的部分保留，把"在边界外侧"的部分丢弃，并在多边形边与边界相交的地方添加交点。

可以想象成用 4 把剪刀，一把一把地修剪纸张——每把剪刀只剪掉纸张在某个方向上的多余部分。

### 为什么逐边裁剪是正确的？

这个问题的直觉解释是：**裁剪是一个可分解的运算**。如果我们把裁剪窗口看作 4 个半平面的交集：

```
ClipWindow = LeftHalfPlane ∩ TopHalfPlane ∩ RightHalfPlane ∩ BottomHalfPlane
```

那么：

```
Polygon ∩ ClipWindow = Polygon ∩ (L ∩ T ∩ R ∩ B)
                     = (((Polygon ∩ L) ∩ T) ∩ R) ∩ B
```

因为交集运算是**可结合**的（associative），所以逐个半平面裁剪与一次性裁剪到矩形的结果是相同的。这个性质是 Sutherland-Hodgman 正确性的数学基础。

> **为什么必须是交集（∩）而不是并集？** 因为裁剪窗口的定义是"四个半平面的共同区域"——一个点必须在所有 4 个半平面内，才算在窗口内。如果用并集（∪），那就变成了"在任意一个半平面内就算在窗口内"——这对应的是窗口之外的区域。

### 要解决的子问题：沿一条边裁剪

在每次裁剪中，我们面对的子问题是：给定一条裁剪边界线和一个多边形，输出位于边界**内侧**的部分。

**第一步：定义"内侧"**

对于矩形裁剪窗口，每条边的"内侧"定义如下（窗口为 [xmin, xmax] × [ymin, ymax]）：

- **左边界**（x = xmin，向上走）：内侧是 x ≥ xmin（右边）
- **上边界**（y = ymax，向右走）：内侧是 y ≤ ymax（下边）
- **右边界**（x = xmax，向下走）：内侧是 x ≤ xmax（左边）
- **下边界**（y = ymin，向左走）：内侧是 y ≥ ymin（上边）

有了这个定义，我们就可以用一个通用的"点是否在边界内侧"的判断函数。

**第二步：判断点在边界的内侧还是外侧——叉积法**

使用**叉积**（Cross Product）来判断。给定一条有向边 E = P₁→P₂，和一个测试点 P：

```
cross = (P₂.x - P₁.x) × (P.y - P₁.y) - (P₂.y - P₁.y) × (P.x - P₁.x)
```

叉积的几何意义（右手定则）：

- `cross > 0` → 点 P 在有向边的**左侧**（从 P₁→P₂ 方向看）
- `cross < 0` → 点 P 在有向边的**右侧**
- `cross = 0` → 点 P 在边的直线上

如果我们让裁剪矩形的 4 条边都按**顺时针**方向排列（左→上→右→下），那么"内侧"就在每条边的**右侧**——即 `cross ≤ 0` 表示在内部。

为什么顺时针时内侧在右侧？想象你站在矩形的边上，顺时针走——你面向的方向变化是"沿墙走"，你的右手永远指向矩形内部。

**第三步：处理四种顶点情况**

对于多边形的每条边 S→E（Start → End），根据两个端点在边界内外的位置关系，共有 4 种情况：

```
情况 1：S 在内，E 在内 → 输出 E
情况 2：S 在内，E 在外 → 输出交点 I（不输出 E）
情况 3：S 在外，E 在内 → 输出交点 I 和 E
情况 4：S 在外，E 在外 → 不输出任何顶点
```

图示（ASCII 艺术）：

```
    内侧                外侧
    ═══════════════════ 裁剪边界
    S─────→E           S─────→I
    │ 输出 E（只输出终点）  │ 输出 I
    │                    │
    情况1              情况2

    I─────→E           S─────→E（两者都在外）
    │ 输出 I+E         │ 无输出（跳过）
    │                   │
    情况3              情况4
```

### 交点计算

当一条多边形边与裁剪边界相交时，需要计算交点。使用**参数化方法**：

给定线段 S(x₁,y₁)→E(x₂,y₂) 和边界线段 B₁(x₃,y₃)→B₂(x₄,y₄)：

```
denom = (x₁-x₂)(y₃-y₄) - (y₁-y₂)(x₃-x₄)
t     = ((x₁-x₃)(y₃-y₄) - (y₁-y₃)(x₃-x₄)) / denom

交点 = S + t × (E - S)
```

其中 t 是线段 SE 上的参数。t=0 是 S 点，t=1 是 E 点，0<t<1 表示交点在两点之间。

为什么用参数化而不是解直线方程？参数化方法更稳定——它直接基于线段的向量表示，避免了斜率为无穷大的特殊情况（垂直的裁剪边界）。

### 数值稳定性考虑

在实际实现中，需要注意三个关键点：

1. **浮点精度**：使用 1e-12 作为判断零的阈值，而不是严格的 0。浮点运算会产生微小误差。
2. **边界情况**：点在边界线上时算内侧还是外侧？必须保持一致——我们的做法是 `cross ≤ 1e-12` 算内侧（包含边界上的点）。
3. **退化多边形**：裁剪后可能得到空多边形（完全在窗外）或退化为只有 1-2 个顶点的线段状多边形——需要验证代码能正确处理这些情况。

### 一句话总结原理

> Sutherland-Hodgman 将 N 边形裁剪到矩形窗口的问题分解为 4 次半平面裁剪，每次裁剪遍历多边形的所有边，根据两端点的内外状态（4 种情况）决定输出哪些顶点，其中内外判断使用叉积（顺时针边内侧在右侧），交点使用参数化线段求交。

## 实现架构

### 整体数据流

```
输入多边形（顶点列表 vector<Point>）
       │
       ▼
  clipEdge(多边形, 左边界)  ← inside() 叉积判断 + intersect() 参数化求交
       │ 输出裁剪后的顶点列表
       ▼
  clipEdge(多边形, 上边界)
       │
       ▼
  clipEdge(多边形, 右边界)
       │
       ▼
  clipEdge(多边形, 下边界)
       │
       ▼
  裁剪后多边形（顶点列表）
       │
       ▼
  可视化渲染（Bresenham 直线 + 扫描线填充）→ PPM 图像
       │
       ▼
  量化验证（面积对比、像素统计、边界检查）
```

数据流的核心特征是**流水线式处理**：每次 clipEdge 的输出直接成为下一次的输入。这种设计天然适合 GPU 中的管线架构——每个裁剪阶段可以并行处理。

### 关键数据结构与设计理由

```cpp
// 二维点——最简单的数据表示
struct Point {
    double x, y;
};
```

**为什么用 double 而不是 float？** 浮点精度在裁剪累加中会放大误差。经过 4 次裁剪，float 的 7 位精度可能导致交点偏离边界线（出现肉眼不可见但面积计算能检测到的误差）。double 的 15 位精度提供了足够的安全裕度。

**为什么不用更复杂的数据结构？**

- Sutherland-Hodgman 算法本身不依赖复杂的数据结构——它只在乎顶点的顺序
- `std::vector<Point>` 天然适合流水线式裁剪：push_back 是 O(1) 摊销
- 每次裁剪的输出作为下一次的输入，内存复用的开销可控

### 函数职责划分

| 函数 | 职责 | 输入 | 输出 | 复杂度 |
|------|------|------|------|--------|
| `inside(p, e1, e2)` | 叉积判断点是否在边界内侧 | 测试点、边界两端点 | bool | O(1) |
| `intersect(s, e, b1, b2)` | 参数化求线段与边界交点 | 线段两端点、边界两端点 | Point | O(1) |
| `clipEdge(poly, e1, e2)` | 沿一条边界裁剪多边形 | 多边形顶点列表、边界 | 新多边形顶点列表 | O(n) |
| `sutherlandHodgman(poly, x/y min/max)` | 主函数：4 次调用 clipEdge | 多边形、矩形参数 | 最终裁剪结果 | O(n) |

### 可视化管线的设计

可视化部分与裁剪算法完全解耦：

1. **Bresenham 直线算法**：绘制多边形边框，无浮点运算、纯整数——保证了边框绘制的像素精确性
2. **扫描线填充算法**：填充多边形内部，支持透明度混合（alpha 通道模拟）
3. **颜色分层策略**：
   - 蓝色半透明（alpha≈0.3）= 原始多边形 → 展示"裁剪前"的范围
   - 红色半透明（alpha≈0.6）= 裁剪后多边形 → 展示"裁剪后"的结果
   - 白色线框 = 裁剪窗口 → 展示裁剪边界
   - 深蓝灰背景 = 窗外区域 → 提供视觉对比
4. **输出格式**：PPM P6 二进制——不需要任何外部图像库，直接用 fstream 写入

## 关键代码解析

### 1. inside() 函数——叉积判断内外

```cpp
bool inside(const Point& p, const Point& edge_p1, const Point& edge_p2) {
    double cross = (edge_p2.x - edge_p1.x) * (p.y - edge_p1.y) 
                 - (edge_p2.y - edge_p1.y) * (p.x - edge_p1.x);
    // 顺时针边：内侧在右侧（cross ≤ 0）
    return cross <= 1e-12;
}
```

**为什么用 1e-12 而不是 0？**

浮点数有精度损耗。如果点在边界线上，叉积理论上为 0，但浮点计算可能得到 ±1e-15。使用 `≤ 1e-12` 而非严格的 `≤ 0`：
- 将 ±1e-15 的微小误差当作"在边界上"处理
- 1e-12 远大于浮点误差（~1e-15），远小于有意义的几何区分（像素级 ~1.0）
- 这个阈值经过了实际测试的验证

**顺时针 vs 逆时针的陷阱**

我们定义矩形边为顺时针：
- 左：`(xmin, ymin) → (xmin, ymax)` 向上 ↑
- 上：`(xmin, ymax) → (xmax, ymax)` 向右 →
- 右：`(xmax, ymax) → (xmax, ymin)` 向下 ↓
- 下：`(xmax, ymin) → (xmin, ymin)` 向左 ←

顺时针意味着"内侧"在有向边的右侧（cross ≤ 0）。如果误用逆时针，必须改为 `cross ≥ -1e-12`（内侧在左侧）。

**常见的 Bug：符号反了**

第一次实现时，我写成了 `cross >= -1e-12`（认为内侧在左侧），结果所有裁剪结果都是空多边形——因为所有顶点都被判为"在边界外侧"。这个 Bug 在视觉上表现为"输出图片中只有原始蓝色多边形，红色裁剪结果区域为零"。

### 2. intersect() 函数——参数化求交点

```cpp
Point intersect(const Point& p1, const Point& p2,
                const Point& edge_p1, const Point& edge_p2) {
    double x1 = p1.x, y1 = p1.y;
    double x2 = p2.x, y2 = p2.y;
    double x3 = edge_p1.x, y3 = edge_p1.y;
    double x4 = edge_p2.x, y4 = edge_p2.y;

    double denom = (x1 - x2) * (y3 - y4) - (y1 - y2) * (x3 - x4);
    if (std::abs(denom) < 1e-12) {
        return p2;  // 平行情况下的安全回退
    }

    double t = ((x1 - x3) * (y3 - y4) - (y1 - y3) * (x3 - x4)) / denom;
    t = std::max(0.0, std::min(1.0, t)); // 钳制到 [0, 1]

    return Point(x1 + t * (x2 - x1), y1 + t * (y2 - y1));
}
```

**计算原理**：

两条线段的参数化表示：
- 线段 1：P₁ + t × (P₂ - P₁)，t ∈ [0, 1]
- 线段 2：E₁ + u × (E₂ - E₁)，u ∈ [0, 1]

联立方程 `P₁ + t(P₂-P₁) = E₁ + u(E₂-E₁)`，得到关于 t 和 u 的 2×2 线性方程组。我们只需要 t（交点在多边形边上的位置），因为边界的位置不关键。

**为什么钳制 t 到 [0, 1]？**

浮点误差可能导致 t 略小于 0 或略大于 1。如果不钳制：
- t = -0.00000001 → 交点出现在线段起点的外侧，裁剪结果会多出不应有的顶点
- t = 1.00000001 → 交点出现在线段终点的外侧，同样的问题

钳制确保了交点始终在多边形边的实际范围内。

**为什么平行时返回 p2 而不是报错？**

这种情况在实际裁剪中极罕见——只有当多边形的一条边恰好与裁剪边界平行时才发生。但作为防御性编程，返回 p2 是安全的默认选择：
- 如果边平行且端点在边界上，p2 本身就在正确的位置
- 如果边平行且端点不在边界上，该情况4会被跳过，这里返回的值不会进入输出

### 3. clipEdge() 函数——核心裁剪逻辑

```cpp
std::vector<Point> clipEdge(const std::vector<Point>& polygon,
                            const Point& edge_p1, const Point& edge_p2) {
    std::vector<Point> output;

    for (size_t i = 0; i < polygon.size(); i++) {
        const Point& current = polygon[i];
        const Point& next = polygon[(i + 1) % polygon.size()];

        bool curr_inside = inside(current, edge_p1, edge_p2);
        bool next_inside = inside(next, edge_p1, edge_p2);

        if (curr_inside && next_inside) {
            // 情况1：两端都在内侧 → 只输出终点
            output.push_back(next);
        } else if (curr_inside && !next_inside) {
            // 情况2：起点在内，终点在外 → 只输出交点
            output.push_back(intersect(current, next, edge_p1, edge_p2));
        } else if (!curr_inside && next_inside) {
            // 情况3：起点在外，终点在内 → 输出交点 + 终点
            output.push_back(intersect(current, next, edge_p1, edge_p2));
            output.push_back(next);
        }
        // 情况4：两端都在外侧 → 不输出（隐式跳过）
    }

    return output;
}
```

**为什么情况1只输出终点？**

这是 Sutherland-Hodgman 设计中最巧妙的细节。考虑遍历过程：
- 对于多边形边 S₀→S₁：如果 S₁ 在内侧，输出 S₁（此时 S₁ 已添加到输出列表）
- 对于下一条边 S₁→S₂：从 S₁ 开始，S₁ 已经在输出列表中（作为上一条边的终点）

因此，每次处理"两端都在内侧"的情况时，只需要输出**终点**——起点已经由前一条边负责输出了。这避免了顶点重复。

**为什么情况2只输出交点？**

当从内侧走到外侧时，多边形正在"离开"裁剪区域。裁剪后的边界应该在**交点处截断**，然后裁剪后的多边形等待下一个从外到内的边（情况3）来"重新进入"。

**为什么用 `(i+1)%polygon.size()`？**

多边形是闭合的——最后一条边必须从最后一个顶点连回第一个顶点。取模运算自然地实现了这个闭圈。如果不做这一步，多边形会缺失最后一条边（顶点N-1 到 顶点0 的边），导致裁剪窗口边缘出现一个"缺口"。

### 4. 扫描线填充与 Bresenham 直线

填充多边形使用标准的**扫描线算法**：

```cpp
void fillPolygon(/* buffers */, const std::vector<Point>& poly,
                 uint8_t cr, uint8_t cg, uint8_t cb,
                 int width, int height, double alpha) {
    // 1. 找到多边形的 y 范围
    double ymin = poly[0].y, ymax = poly[0].y;
    for (const auto& p : poly) {
        ymin = std::min(ymin, p.y);
        ymax = std::max(ymax, p.y);
    }

    // 2. 对每一行扫描线
    for (int y = ymin; y <= ymax; y++) {
        // 3. 找扫描线与所有多边形边的交点
        std::vector<double> x_intersections;
        for (size_t i = 0; i < poly.size(); i++) {
            const Point& p1 = poly[i];
            const Point& p2 = poly[(i + 1) % poly.size()];
            // 判断扫描线是否穿过这条边（排除水平边）
            if ((p1.y <= y && p2.y > y) || (p2.y <= y && p1.y > y)) {
                double t = (y - p1.y) / (p2.y - p1.y);
                double x = p1.x + t * (p2.x - p1.x);
                x_intersections.push_back(x);
            }
        }

        // 4. 排序交点，成对填充
        std::sort(x_intersections.begin(), x_intersections.end());
        for (size_t i = 0; i + 1 < x_intersections.size(); i += 2) {
            // 绘制第 i 个到第 i+1 个交点之间的水平线段
        }
    }
}
```

**扫描线填充的正确性依赖于多边形是闭合的**——如果多边形不自交，扫描线与多边形的交点总是成对出现（进入→离开→进入→离开...），所以成对填充是正确的。

**alpha 混合的处理**：在像素写入前进行线性插值，实现半透明效果。这使得我们可以在同一张图上叠放原始多边形（蓝色半透明）和裁剪后多边形（红色半透明），且重叠区域呈现紫色——方便视觉对比。

## 踩坑实录

### 坑 1：叉积方向搞反了——所有裁剪结果为空

**症状**：代码编译通过、运行不崩溃，但 4 个测试用例的裁剪结果全是空多边形（Vertex Change 列显示 `→ 0`）。视觉上只有蓝色原始多边形可见，红色裁剪区域不存在。

**错误假设**：`cross >= 0` 表示"在边界内侧"。我最初认为顺时针边中"内侧在左侧"——这是错误的。

**真实原因**：叉积的几何含义是：在 2D 平面中，`(v1 × v2) > 0` 表示 v2 在 v1 的**逆时针方向**（左侧）。当边界边按顺时针排列时，"内侧"在边的**右侧**（顺时针方向），所以应该是 `cross ≤ 0`。

**具体验证**：
```
左边界: (250, 200) → (250, 400) 向上走
测试点: (400, 300) — 窗口中心，应该在内部
cross = (250-250)*(300-200) - (400-250)*(400-200) 
      = 0 - 150*200 = -30000
      → cross < 0 才是内部！
```

**修复方式**：将 `cross >= -1e-12` 改为 `cross <= 1e-12`。

**教训**：在图形学中，叉积方向是出错的"重灾区"。每次使用前应该用具体数值验证——选一个已知内外的点，手算一遍叉积。

### 坑 2：编译错误——变量名遮蔽

**症状**：编译报错 `invalid initialization of reference`，指向 drawLine 调用。

**错误假设**：代码逻辑没问题，编译参数没问题。

**真实原因**：在 main() 函数中，for 循环的迭代变量 `a`、`b` 是 `Point` 类型，但 drawLine 的参数中也用到了名为 `b` 的 buffer 引用（`std::vector<std::vector<uint8_t>>& b`）。C++ 的名称查找规则将 `b` 解析为最内层作用域的变量——即 `Point b` 而非 buffer `b`，导致类型不匹配。

**修复方式**：将循环变量改名为 `pa`、`pb`，避免遮蔽。

**教训**：单字母变量名在大型函数中容易引发名称冲突。特别是在需要多层嵌套循环且上层变量是引用类型时，更应该使用有区分度的名称。

### 坑 3：PNG 文件太小

**症状**：量化验证要求文件大小 > 10240 字节，但 800×600 输出的 PNG 只有 10184 字节——仅差 56 字节。

**原因**：800×600 分辨率对 PNG 压缩来说刚好接近阈值。图片大部分是纯色背景（深蓝色），压缩率极高。

**修复方式**：提升分辨率到 1200×900，PNG 文件增至 14420 字节（增幅 41%）。

**教训**：量化阈值不能设置得太临界。在实际项目中，应该根据内容复杂度来设定合理阈值，或者使用"PNG + PPM 双重检查"。

### 坑 4：凹多边形测试用例偏移后进入窗外

**症状**：将裁剪窗口从 (250,200)-(550,400) 改为 (400,300)-(800,600) 后，凹多边形裁剪结果面积接近 0。

**原因**：凹多边形的坐标是硬编码的（220,150）等，在窗口大小和位置改变后，这些坐标完全落到了窗外。

**修复方式**：重新设计测试用例坐标，使其始终与窗口参数保持相对关系。更好的做法是将测试多边形用窗口中心的百分比坐标定义，然后动态计算绝对坐标。

### 算法正确性验证逻辑

我们使用了 4 层量化检查（全部自动化、无人工判断）：

1. **边界约束检查**：裁剪后所有顶点必须在 `[xmin, xmax] × [ymin, ymax]` 内
2. **面积单调性检查**：裁剪后面积 ≤ 原始面积 ≤ 裁剪窗口面积
3. **非空性检查**：如果原始多边形有顶点在窗内，裁剪结果不能为空
4. **像素统计检查**：图像不能全黑（mean<5）或全白（mean>250），必须有足够内容变化（std>5）

## 总结与延伸

### 技术局限性

1. **只能处理凸裁剪窗口**：Sutherland-Hodgman 假设裁剪窗口是凸的。对于凹裁剪窗口（如 L 形或星形窗口），结果可能不正确——逐半平面裁剪会将凹区域"凸化"。

2. **无法处理自交多边形**：如果输入多边形自交（像蝴蝶结形状），Sutherland-Hodgman 的输出可能是退化的——算法将它当作简单多边形处理，自交区域的"内外"判断会出错。

3. **无法生成空洞多边形**：裁剪结果总是简单多边形。如果原始多边形的一个洞部分在窗口内，算法无法表示这种"空心"结构。

4. **多次裁剪的浮点误差累积**：4 次 clipEdge 调用中，每次的交点计算都会引入微小的浮点误差。对于顶点非常多（N>1000）的多边形，误差累积可能导致不可忽略的视觉瑕疵。

### 优化方向

1. **添加 Weiler-Atherton 实现**：扩展到支持任意形状的裁剪窗口，处理凹窗口和带洞多边形
2. **SIMD 加速**：利用 AVX2/SSE 指令并行计算多个顶点的叉积判断
3. **GPU 实现**：使用 Compute Shader 并行化逐边裁剪
4. **自适应精度**：根据多边形规模动态选择 float/double，平衡精度和性能
