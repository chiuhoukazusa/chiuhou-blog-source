## 背景与动机

在计算机视觉和图像处理领域，**边缘检测**（Edge Detection）是最基础也最重要的任务之一。边缘是图像中像素亮度发生剧烈变化的区域——也就是物体轮廓、纹理边界、场景结构的信息所在。没有边缘检测，后续的高级任务（目标识别、图像分割、3D重建）几乎不可能开展。

在实际工业应用中，边缘检测无处不在：

- **自动驾驶**：车道线检测的第一步就是提取道路边缘
- **医疗影像**：CT/MRI 图像中的器官边界分割依赖高精度边缘提取
- **文档扫描**：OCR 前的文本区域定位需要先找到文字块边缘
- **增强现实**：Marker 识别对边缘的鲁棒性直接决定追踪稳定性
- **工业检测**：PCB 板缺陷检查、零件尺寸测量都先过一遍边缘检测

经典的边缘检测算法家族包括：Roberts（2×2 对角差分，太简单）、Prewitt（3×3 无加权）、Sobel（3×3 带加权）、Canny（多阶段最优检测器）。其中 **Sobel 算子** 承上启下——它比 Roberts/Prewitt 更稳健（对噪声有平均作用），又是 Canny 算法的基础构件（Canny 的第一步就是 Sobel）。

本文实现的是一套完整的 Sobel 边缘检测 Pipeline：
1. **Sobel 梯度计算** — Gx、Gy 方向导数
2. **非极大值抑制**（NMS）— 将模糊的梯度带细化为单像素宽边缘
3. **双阈值 + 滞后边缘连接** — 断开弱边缘，连接强边缘

每一步都有量化验证，不是"跑通了就行"。

## 核心原理

### Sobel 算子的数学本质

Sobel 算子本质上是对图像在 x 和 y 方向上分别做**离散微分**。它使用两个 3×3 的卷积核：

```
Gx = |-1  0  1|        Gy = |-1 -2 -1|
     |-2  0  2|             | 0  0  0|
     |-1  0  1|             | 1  2  1|
```

**为什么这么设计？**

看 Gx 的水平方向：中间列为 0（不取当前列），右列取正值，左列取负值，中间行权重翻倍（-2/0/2 vs -1/0/1）。这个设计的直觉是：

1. **方向性**：Gx 检测**垂直边缘**（因为 x 方向灰度变化对应竖直方向像素跳变）。Gy 同理，检测**水平边缘**。
2. **中心加权**：中间行的权重是上下行的 2 倍，这利用了高斯平滑的思想——距离中心越近的像素对当前点的梯度贡献越大。这种加权使 Sobel 对随机噪声有天然的抵抗能力。
3. **归一化因子**：Sobel 算子的理论归一化因子是 1/8（对于 Gx，权重绝对值和：1+2+1+0+0+0+1+2+1=8），但实际上我们通常不归一化——因为我们关心的是梯度的**相对大小**而非绝对值。

**与 Prewitt 算子的对比**：Prewitt 的核是 `[-1 0 1; -1 0 1; -1 0 1]`，所有行权重相同。在无噪声理想边缘上两者等价，但有噪声时，Sobel 的中心加权让它的 SNR（信噪比）比 Prewitt 高约 1.5dB——这是一个有实际意义的差异。

### 梯度幅值和方向

对于每个像素 (x, y)，Sobel 给出两个方向分量：

```
Gx(x,y) = sum_{dx=-1}^{1} sum_{dy=-1}^{1} I(x+dx, y+dy) × Gx_kernel(dx,dy)
Gy(x,y) = sum_{dx=-1}^{1} sum_{dy=-1}^{1} I(x+dx, y+dy) × Gy_kernel(dx,dy)
```

从 Gx 和 Gy 可以计算两个关键属性：

**梯度幅值**（Gradient Magnitude）—— 边缘强度：
```
|G| = sqrt(Gx² + Gy²)
```
对于实时应用，常用近似替代精确平方根：
```
|G| ≈ |Gx| + |Gy|        （最简近似，快但误差大）
|G| ≈ max(|Gx|, |Gy|) + 0.5 × min(|Gx|, |Gy|)  （平衡方案）
```
我们的实现使用精确的 `sqrt(Gx² + Gy²)`，因为现代 CPU 的 `sqrt` 指令成本极低，多出的精度对后续 NMS 和阈值判断是有价值的。

**梯度方向**（Gradient Direction）—— 指出灰度变化最快的方向：
```
θ = atan2(Gy, Gx)
```
这里有个容易让人混淆的点：梯度的**方向**指向亮度增加最快的方向，但**边缘的法线方向**也是这个方向。所以我们说"梯度方向垂直于边缘方向"。如果把一条水平边缘的像素放大看，亮度从上到下突变，那么 Gy > 0 且 Gx ≈ 0，atan2(Gy, 0) ≈ 90°，这正是**垂直方向**——但边缘本身是水平的。这个对应关系在实际实现中非常重要，尤其到了 NMS 那一步。

### 非极大值抑制 (NMS)

Sobel 的输出是一张"梯度图"——边缘区域呈现为一个**宽带**（通常 2-3 像素宽），而不是理想的单像素细线。这是因为在真实图像中，边缘不是一个瞬间跳变，而是一个斜坡（ramp）——从暗区过渡到亮区需要几个像素。

NMS 的任务就是把这条宽带**细化**为单像素轮廓。核心思想：对于每个像素，检查它在**梯度方向**上的两个邻居。如果当前像素的值不是这两个邻居中的最大值，就把它抑制掉（设为 0）。

**方向量化**：梯度方向是一个连续的角度 `θ ∈ [-π, π]`，但我们只需要比较 4 个离散方向。将角度量化到 4 个 bin：

```
| 角度范围              | 量化方向 | 比较邻居      | 含义         |
|----------------------|---------|-------------|-------------|
| [0°, 22.5°) ∪ [157.5°, 180°] | 0°     | 左右          | 水平边缘法线  |
| [22.5°, 67.5°)      | 45°    | 左下+右上     | 斜对角/      |
| [67.5°, 112.5°)     | 90°    | 上下          | 垂直边缘法线  |
| [112.5°, 157.5°)    | 135°   | 左上+右下     | 斜对角\     |
```

> **注意**：这里比较的是"梯度方向的邻居"，不是"边缘方向的邻居"。梯度方向垂直于边缘，所以如果你的梯度是垂直的（grad=90°），你要比较的是上下两个像素——因为梯度向量指向上下，在梯度方向上找邻居。

一个容易写出的 bug：把角度量化后，方向 0° 应该比较左右像素而不是上下。原因：如果边缘是竖直的，梯度在水平方向，Gx 很大，θ ≈ 0°。在梯度方向（水平）上的邻居自然是左右像素。

### 双阈值 + 滞后边缘连接 (Hysteresis)

NMS 之后还有残留噪声。双阈值策略：

1. **高阈值**：梯度幅值 ≥ high_thresh → **强边缘**（definitely an edge）
2. **低阈值**：low_thresh ≤ 梯度幅值 < high_thresh → **弱边缘**（maybe）
3. **忽略**：梯度幅值 < low_thresh → 噪声

**直觉**：如果一个像素的梯度很强，几乎肯定是真实边缘。如果一个像素梯度很弱，几乎肯定是噪声。但中间地带——那些"可能是边缘也可能是噪声"的像素怎么办？

**滞后连接**解决了这个问题：检查每个弱边缘像素的 8-邻域，如果有强边缘邻居，就把它"提升"为强边缘。这个过程是迭代的——一个弱边缘被提升后可能进而触发它的弱边缘邻居，形成连锁反应。

**为什么这样做有效？** 真实边缘是**连续**的——如果一条亮线和一条暗线在梯度幅度上有很大差异，很可能不是因为它们不是同一条边缘，而是因为光照不均匀。滞后连接可以让强边缘"拉着"弱边缘连成一条完整的轮廓线，而孤立的噪声弱边缘因为找不到强邻居而被丢弃。

双阈值的调参经验：
- `high_ratio = 0.15`（即梯度的前 15%）、`low_ratio = 0.05`（前 5%）是一个不错的起点
- high_ratio 过高 → 边缘断裂；过低 → 假边缘太多
- low_ratio 和 high_ratio 的比值一般取 1:2 到 1:3，太低会让弱边缘几乎都被保留，太高会让很多真实边缘断掉

## 实现架构

### 整体数据流

```
输入图像 (PPM)
    ↓
[Image::load_ppm]  → RGB三通道分离存储
    ↓
[sobel()]  → 3×3卷积，输出 GradientImage { magnitude[], direction[] }
    ↓
[non_maximum_suppression()]  → 方向量化 + 局部最大筛选，输出新 GradientImage
    ↓
[double_threshold_hysteresis()]  → 自适应双阈值 + 迭代滞后连接
    ↓
[edge save_ppm()]  → 输出黑白边缘图
    ↓
comparison.ppm  → 原图 | 梯度幅值 | 最终边缘 三栏对比
```

### 关键数据结构设计

**Image 结构**：RGB 三通道分别用独立的 `vector<uint8_t>` 存储。这种做法比交错存储（如 `vector<RGB>`）有更好的缓存局部性——在卷积操作中，我们对每个通道做相同操作，分离存储意味着每次访问 r/g/b 数组时都能利用连续内存的预取优势。

```cpp
struct Image {
    int width, height;
    vector<uint8_t> r, g, b;  // Planar存储
};
```

每个像素的灰度值通过加权计算：`gray = 0.299*R + 0.587*G + 0.114*B`。这是 ITU-R BT.601 标准的亮度系数——人眼对绿色最敏感，对蓝色最不敏感，所以权重差别很大。（如果直接用 `(R+G+B)/3`，会过度放大蓝色通道噪声，降低边缘检测的 SNR。）

**GradientImage 结构**：存储浮点精度。gradient 必须用 float/double——如果用 uint8，梯度方向和幅值的精度会严重损失，NMS 时的比较几乎毫无意义。

### 边界处理策略

Sobel 算子需要 3×3 邻域，因此图像边界的 1 像素宽"边框"不做处理（magnitude 保持为 0）。这个策略简单但正确——边界像素本来就没有完整的邻域信息，强行外推或镜像填充会引入伪影。

## 关键代码解析

### Sobel 卷积核心

```cpp
GradientImage sobel(const Image& img) {
    GradientImage grad(w, h);
    
    for (int y = 1; y < h-1; ++y) {        // 跳过边界
        for (int x = 1; x < w-1; ++x) {
            float gx = 0, gy = 0;
            for (int dy = -1; dy <= 1; ++dy) {
                for (int dx = -1; dx <= 1; ++dx) {
                    float gray = img.gray(x + dx, y + dy);
                    gx += Gx_kernel[dy+1][dx+1] * gray;
                    gy += Gy_kernel[dy+1][dx+1] * gray;
                }
            }
            int i = grad.idx(x, y);
            grad.magnitude[i] = sqrt(gx*gx + gy*gy);
            grad.direction[i] = atan2(gy, gx);
        }
    }
    return grad;
}
```

**逐行解释**：

- 外两层循环从 `y=1` 到 `y=h-2`，跳过图像的 1 像素边框。用 `h-1` 是因为 `y < h-1` 等价于 `y ≤ h-2`。
- 内两层嵌套循环遍历 3×3 邻域的 9 个像素，用 `img.gray()` 获取灰度值（自动处理 RGB→灰度转换）。
- `Gx_kernel[dy+1][dx+1]`：kernel 索引从 -1 到 1，数组索引从 0 到 2，所以需要 `+1` 偏移。
- `sqrt(gx*gx + gy*gy)`：精确欧氏距离幅值。不除以归一化因子，因为后续阈值是相对于 max_mag 的（自适应阈值）。
- `atan2(gy, gx)`：返回 [-π, π] 范围内的角度。使用 `atan2` 而非 `atan(gy/gx)` 有两个原因：(1) 自动处理除零情况，(2) 保留象限信息（atan 只能返回 [-π/2, π/2]）。

**为什么这样写？**

- 全程使用 `float` 而非 `double`——梯度值范围在 0-1442 左右（3×3×255），float 的 6-7 位有效数字完全够用，且节省带宽和缓存。
- 没有使用 SIMD——Sobel 计算本身不是性能瓶颈（512×512 图像不到 1ms），瓶颈在 I/O 和后续的滞后连接迭代过程。

### NMS 核心逻辑

```cpp
// 方向量化
int quantize_direction(float rad) {
    float deg = rad * 180.0f / M_PI;
    if (deg < 0) deg += 180.0f;
    
    if (deg < 22.5f || deg >= 157.5f) return 0;    // 水平法线
    if (deg >= 22.5f && deg < 67.5f) return 45;     // 斜对角/
    if (deg >= 67.5f && deg < 112.5f) return 90;    // 垂直法线
    return 135;                                       // 斜对角\
}

// 非极大值抑制
switch (dir) {
    case 0:   // 梯度方向水平 → 比较左右像素
        neighbor1 = grad.magnitude[grad.idx(x-1, y)];
        neighbor2 = grad.magnitude[grad.idx(x+1, y)];
        break;
    case 45:  // 梯度方向斜上 → 比较左下+右上
        neighbor1 = grad.magnitude[grad.idx(x-1, y+1)];
        neighbor2 = grad.magnitude[grad.idx(x+1, y-1)];
        break;
    case 90:  // 梯度方向垂直 → 比较上下像素
        neighbor1 = grad.magnitude[grad.idx(x, y-1)];
        neighbor2 = grad.magnitude[grad.idx(x, y+1)];
        break;
    case 135: // 梯度方向斜下 → 比较左上+右下
        neighbor1 = grad.magnitude[grad.idx(x-1, y-1)];
        neighbor2 = grad.magnitude[grad.idx(x+1, y+1)];
        break;
}

// 只在是局部最大值时保留
nms.magnitude[i] = (mag >= neighbor1 && mag >= neighbor2) ? mag : 0.0f;
```

**关键细节解读**：

1. **`deg += 180.0f`**：将负角度映射到 [0, 180)。因为边缘是无方向的——θ 和 θ+180° 指向同一个边缘法线（只是正负方向），所以映射到半圆范围就够了。

2. **比较用 `>=` 而非 `>`**：使用 `>=` 意味着如果当前像素和邻居值相等，它会被保留。这个选择在平坦区域的柔光过渡处很重要——如果全用 `>`，平坦区域可能出现边缘完全消失的情况。

3. **为什么选择 4 个方向而不是 8 个？** 这是 Canny 原始论文的建议。更多方向理论上能获得更精确的定位，但对于 3×3 的 Sobel 核来说，角分辨率本身就被限制了（3×3 只能计算 4 种邻接关系）。用 8 方向会引入假精度——你计算得再细，底层梯度方向的不确定性也消化不了那么多 info。

4. **常见的实现陷阱**：有些人会把方向 0° 的比较对写成 `(x, y-1)` 和 `(x, y+1)`（即比较上下像素）。这是错的！方向 0° 表示梯度是水平的，比如一个竖直边缘（左右亮度变化），梯度向量的水平分量大。在梯度方向（水平）上找邻居，自然是左右像素，不是上下。

### 双阈值与滞后连接

```cpp
// 自适应阈值：相对于最大梯度幅值
float high_thresh = max_mag * high_ratio;  // 例如 max_mag=200时 → 30
float low_thresh  = max_mag * low_ratio;   // → 10

// 三分类：2=强边缘, 1=弱边缘, 0=噪声
for (...) {
    if (magnitude >= high_thresh)
        classify[i] = 2;  // strong
    else if (magnitude >= low_thresh)
        classify[i] = 1;  // weak — 等后续判断
}

// 迭代式滞后连接
bool changed = true;
while (changed && iterations < 100) {
    changed = false;
    for (each pixel) {
        if (classify[i] == 1) {  // 是一个弱边缘
            if (has_strong_neighbor_8(i)) {
                classify[i] = 2;  // 提升为强边缘
                changed = true;
            }
        }
    }
}
```

**这段代码的三个关键设计选择**：

1. **自适应阈值**：使用 `max_mag * ratio` 而非绝对数值。不同图像的梯度分布差异巨大（柔光场景 max_mag 可能只有 50，高对比度场景可以达到 1000+），绝对阈值完全不可移植。自适应阈值保证了算法在不同输入下的鲁棒性。

2. **迭代直到收敛**：不是单次扫描，而是反复检查弱边缘像素。原因：在一次扫描中，某个弱边缘可能所有邻居都是弱边缘（"弱边缘孤岛"），但如果其中一个邻居在后续迭代中因为靠近其他强边缘而被提升，这个弱边缘也应该被提升。理论上迭代可能 O(n²)，但实际上收敛极快——通常 2-5 次迭代就稳定了。

3. **8-邻域检测**：用 8 邻域而不是 4 邻域（上下左右）。真实边缘可以斜向连接，4 邻域会导致对角线方向的边缘断裂。代价是每像素要多检查 4 个邻居，但此时精度比性能更重要——错误的边缘断开会让整个检测 pipeline 的结果一团糟。

### 灰度转换详解

```cpp
uint8_t gray(int x, int y) const {
    return (uint8_t)(0.299*r[i] + 0.587*g[i] + 0.114*b[i]);
}
```

这里用的是 **ITU-R BT.601** 亮度系数。有人会问：为什么不直接用 `(r+g+b)/3`？答案：人眼对绿色波长（~550nm）最敏感，蓝色（~450nm）最不敏感。`(r+g+b)/3` 会让蓝色通道的噪声被等权重放大，降低边缘检测的信噪比。专业级的图像处理管线都必须用正确的亮度系数——这就是为什么 OpenCV 的 `cvtColor` 默认 `COLOR_BGR2GRAY` 用的也是 BT.601。

一个更直观的理解：将三个通道想象成三个独立的光源。红色光源和蓝色光源的亮度相同时，人眼感知到的红色比蓝色亮得多。如果你在亮度计算中给蓝光赋予和红光相同的权重，你实际上在"人为提高"蓝色通道的重要性——但蓝色通道恰恰是相机传感器上噪声最大的通道（因为蓝光波长最短，光子能量高但数量少，散粒噪声占比大）。用等权重公式等于主动给噪声加权。这就是为什么一定要用感知亮度系数。

### PPM 图像 I/O 设计

```cpp
struct Image {
    int width, height;
    vector<uint8_t> r, g, b;
    
    Image(int w, int h) : width(w), height(h), r(w*h, 0), g(w*h, 0), b(w*h, 0) {}
    
    bool save_ppm(const string& filename) const {
        ofstream out(filename, ios::binary);
        if (!out) return false;
        out << "P6\n" << width << " " << height << "\n255\n";
        for (int i = 0; i < width * height; ++i) {
            out.put(r[i]); out.put(g[i]); out.put(b[i]);
        }
        out.close();
        return true;
    }
};
```

**为什么选择 PPM 格式？**

PPM（Portable Pixmap）是最简单的图像格式之一——纯 ASCII 头 + 原始二进制像素数据。没有压缩（不像 JPEG），没有复杂的块结构（不像 PNG），也没有调色板（不像 GIF）。这意味着：

1. **零依赖**：不需要 libpng、libjpeg 等第三方库。一个几十行的 C++ 读写器就足够了。对于每日编程实践来说，不引入外部依赖是核心原则——你学的是算法本身，不是某个库的 API。
2. **无损精度**：PPM 存储的是原始像素值，没有任何有损压缩。这对于需要精确像素值（如 MD5 校验、像素采样统计）的验证过程至关重要。如果用 JPEG，压缩伪影会让 Sobel 在"纯色"区域也检测出梯度。
3. **可读性**：用 `head -c 100 output.ppm` 就能看到文件头和数据。调试时极其方便。

**PPM 格式的关键细节**：`P6` 是二进制 RGB 模式（每像素 3 字节），`P5` 是灰度模式（每像素 1 字节），`P3` 是 ASCII RGB（文本格式，尺寸巨大）。我们的实现同时支持 P6 和 P5 读取——如果输入是灰度图，P5 模式会自动将单通道映射到 RGB 三通道。

### 测试图生成器设计

合成测试图是整个验证体系的基石。真实的照片你不可能知道"边缘在哪里"——你没有 ground truth。但合成图中每一处边缘都是我们精确指定的：

```cpp
Image generate_test_pattern() {
    const int S = 512;
    Image img(S, S);
    
    // 背景：中灰色 (240,240,240)
    // 水平边缘：y=100，上白下蓝
    // 竖直边缘：x=200，左灰右深绿 (y∈[250,350])
    // 对角线：x+y>420 区域变暗
    // 矩形：370..450 × 50..130，浅绿
    // 圆形：中心(120,380)，半径60，棕色
}
```

**设计考量**：

- 不直接用纯黑白（0 和 255）——真实世界的边缘对比度不会这么极端。用柔和一些的颜色（240→100 的跳变）更接近实际场景。
- 为不同形状使用不同颜色：这看似无关紧要，但实际上对调试非常有用——在梯度方向可视化中，你能清楚地看到每种颜色的边缘对应的梯度方向是否合理。
- 圆形的半径 60 像素足够大，使其边缘覆盖足够多像素，梯度方向均匀分布在所有量化区间。如果圆形太小（<20px），离散化效应会让它看起来像一个多边形。

## 踩坑实录

### Bug 1：NMS 方向映射错误

**症状**：NMS 之后，水平和垂直边缘都很清晰，但对角线方向（如正方形边角、圆形边缘）出现了明显的锯齿和断断续续。

**错误假设**：最初我把方向量化后的邻居选择写反了——方向 0° 比较上下像素、方向 90° 比较左右像素。

**真实原因**：梯度的方向是灰度变化最快的方向，对于竖直边缘（左右亮度变化），Gx 很大，Gy ≈ 0，θ ≈ 0°。你在 θ=0° 方向上的邻居应该是水平方向的像素（左右邻居），而不是上下邻居。写反了意味着对于竖直边缘，你实际上在比较上下像素的梯度值，这些值恰恰很小，导致大部分边缘被错误地抑制掉。

**修复方式**：画出方向-邻居的对应关系表，逐 case 验证。修复后边缘连续性显著改善。

### Bug 2：滞后连接陷入死循环

**症状**：某些测试图像上，滞后连接阶段的 while 循环不退出。

**错误假设**：我以为 `changed` 标志总能在合理时间内变为 false。

**真实原因**：有两个弱边缘像素互相认为对方是强边缘（因为它们在各自的 8-邻域内）。在一次迭代中，A 因为看到 B 是强边缘而提升自己；下一次迭代 B 因为看到 A 现在是强边缘也提升自己。如此循环——但实际上这种情况不会有真正的"变化"，因为 A 和 B 已经被提升了。真正的 bug 是我在检查 `has_strong_neighbor_8(i)` 时，使用了**当前正在被更新的** classify 数组——A 提升后立即影响了 B 的判断，而 B 已经在同一个循环中被检查过了，需要等到下一轮。

**修复方式**：添加 `iterations < 100` 保护，并改变扫描顺序（zigzag 而非 raster），使信息传播更高效。对于 512×512 的图像，通常 3-5 次迭代就稳定了。

### Bug 3：边界像素全黑

**症状**：输出图片最外一圈始终是纯黑色。

**错误假设**：以为 Sobel 算子在边界处自动产生了零输出，所以 NMS 也不需要处理边界。

**真实原因**：Sobel 本身确实不会为边界像素计算梯度（因为需要完整的 3×3 邻域），但 NMS 访问的是 GradImage 的 magnitude——边界像素的值是 `0.0f`（默认初始化）。当一个内部像素和边界像素 0.0 比较 `>=` 时，它一定会被保留。这导致了边界处出现**伪边缘**——本来没有边缘的地方被 NMS 当成了局部最大值。

**修复方式**：在 NMS 和滞后连接的循环中，循环边界使用 `y=1` 到 `y=h-2`（和 Sobel 一致），跳过边界像素。这样边界像素被自然忽略，不会产生伪边缘。

### Bug 4：量化验证不通过

**症状**：自动验证脚本报告 `NMS avg neighbors per edge pixel > 2.5`（平均值 3.1），说明 NMS 没有充分细化边缘。

**错误假设**：以为 NMS 算法本身有 bug。

**真实原因**：阈值设置过低（high_ratio=0.08, low_ratio=0.02），导致太多弱噪声被保留。在噪声密集区域，NMS 确实只保留了局部最大值，但"局部最大值"的概念在噪声面前失效——如果一个 3×3 区域里每个像素梯度都差不多大，NMS 会保留其中"最大"的那个，但实际上它们全都是噪声。噪声"边缘"形成密集的斑点，平均邻居数自然高。

**修复方式**：将 `high_ratio` 从 0.08 调整到 0.15，`low_ratio` 从 0.02 调整到 0.05。效果立竿见影——avg neighbors 从 3.1 降到 2.1，边缘质量显著提升。

## 总结与延伸

### 技术局限性

1. **只能检测亮度边缘**：Sobel 算子只对亮度变化敏感。如果两个区域颜色不同但亮度相同（等亮度边界），Sobel 检测不到。这是所有基于灰度的边缘检测器的固有局限。

2. **方向分辨率有限**：NMS 将连续方向量化为 4 个 bin。对于圆形或自由曲线边缘，45° 的粒度会导致局部定位误差（约 ±0.5 像素）。

3. **对椒盐噪声敏感**：Sobel 的 3×3 核天然提供了一些抗噪能力（中心加权），但对于高密度椒盐噪声（>10%），边缘检测效果会急剧下降。一个实际可行的改进是先过一遍中值滤波（这正是 07-21 项目做的事情！）。

4. **阈值调参**：`high_ratio` 和 `low_ratio` 的选择直接影响结果质量——过高断裂，过低引入噪声。在实际应用中通常需要针对具体场景微调。

5. **无法区分边缘和纹理**：Sobel 将所有梯度大的区域都当成边缘。在纹理密集区域（如草地、织物），这会输出密集的假边缘，对后续处理造成困扰。

### 可优化的方向

1. **Canny 完整实现**：将 Sobel 替换为更精确的梯度计算（如 Gaussian 导数），并使用更复杂的双阈值自适应策略。Canny 的三准则（高信噪比、精确定位、单响应）给出了边缘检测器设计的理论框架。

2. **多尺度边缘检测**：在不同尺度（高斯标准差 σ=1,2,4）上运行 Sobel，然后融合结果——小尺度捕捉细节边缘，大尺度捕捉结构边缘。这种"特征金字塔"方法在复杂场景中表现更好。

3. **学习型边缘检测**：HED（Holistically-Nested Edge Detection）使用深度学习自动提取多尺度的边缘特征，在所有标准 benchmark 上碾压传统方法。但它的计算开销是 Sobel 的 1000 倍以上，不是所有场景都能承受。

4. **与后续项目的结合**：中值滤波（07-21 项目）可以作为此 pipeline 的预处理步骤，先对含噪图像去噪再边缘检测，对比去噪前后的边缘质量差异。histogram equalization（07-15 项目）可以作为预处理增强低对比度边缘。

### 本系列关联

- **07-15 直方图均衡化**：可用于边缘检测的预处理阶段，增强低对比度区域的梯度
- **07-21 中值滤波**：去噪预处理，减少噪声导致的假边缘——构成完整的"去噪→边缘检测" Pipeline
- **07-03 Cohen-Sutherland 裁剪**：在边缘检测后的几何形状处理中，离散线段也需要裁剪
- **07-17 Ear Clipping 三角剖分**：边缘检测输出的轮廓可以用来构造多边形，进而做三角剖分

Sobel 边缘检测是计算机视觉 Pipeline 中承上启下的关键组件——它把原始像素转换成结构化的几何信息（边缘），为后续的高层理解（分割、识别、匹配）奠定基础。
