## ① 背景与动机

寻路（Pathfinding）是游戏AI的基石功能。从一个RTS游戏里的农民去伐木，到一个开放世界RPG中NPC在城镇里巡逻，再到MOBA游戏里小兵沿兵线前进——所有这些行为的背后都依赖寻路算法来找到从起点到终点的最短路径。

经典的A*算法（1968年提出）是寻路领域的黄金标准。它结合了Dijkstra的最短路径保证和贪心最佳优先搜索的效率，通过一个启发式函数`f(n) = g(n) + h(n)`来引导搜索方向。在均匀网格上，A*使用八方向移动（上下左右+四个对角线），每次将当前节点的八个邻居加入open list，然后从中选择f值最小的节点扩展。

**A*的痛点**：在一个空旷的128×128网格上，从(0,0)寻路到(127,127)，A*需要扩展约3400个节点。这对于即时战略游戏来说不算多，但在需要同时处理数百个单位寻路的场景下（或者网格更大、地图更复杂的场景），A*的节点扩展量会成为性能瓶颈。

**为什么A*会扩展这么多节点？** 在空旷区域中，大多数路径都是对称的。比如从(0,0)到(5,5)，你可以走"右上右上右上右上右上"，也可以走"上右上右上右上右"——这些路径长度相同，启发式函数也不会区分它们，导致A*会探索大量等价路径。

**Jump Point Search (JPS)** 是2011年由Daniel Harabor和Alban Grastien提出的A*优化变体。它的核心洞察是：网格上的大多数节点都是"冗余"的——你不需要逐个检查路径上的每个格子，只需要在路径方向发生变化的地方（即"跳点"）做决策。JPS通过"向前看"的方式跳过中间节点，将搜索空间缩减为仅包含真正需要决策的节点。

**工业应用场景**：
- **游戏引擎**：Unity的A* Pathfinding Project、UE的Navigation System都提供了JPS实现
- **即时战略游戏**：大量单位同时寻路，节点扩展量直接影响帧率
- **大规模网格地图**：2000×2000以上的地图，A*可能扩展数十万节点，JPS可以在一秒内完成
- **移动端游戏**：CPU资源有限，减少节点扩展意味着更低的功耗

JPS的理论保证是：在均匀成本网格地图上，JPS找到的路径与A*找到的路径**具有完全相同的最优代价**，但通常只需要扩展A*的30%-50%的节点数量。

## ② 核心原理

### 2.1 对称路径修剪（Symmetric Path Pruning）

JPS的第一个关键思想是**对称路径修剪**。考虑以下场景：

```
从节点A到节点B，经过节点X：
A → ... → X → ... → B

如果存在另一条路径 A → ... → Y → ... → B，且两条路径长度相同，
那么X就是"对称的"——探索X不能带来任何优势。
```

具体来说，JPS定义了以下规则：

**直线移动时的邻居修剪**（以向右移动为例，即方向向量为(1,0)）：
- 正常A*会检查节点X的所有8个邻居
- JPS只检查3个邻居：
  - **自然邻居**（Natural Neighbor）：正前方(1,0)的节点——这是继续沿着当前方向走所必需的
  - **强制邻居**（Forced Neighbor）：上方(1,-1)和下方(1,1)的节点——仅当它们符合"强制"条件时

**强制邻居的判断条件**：
```
检查上方强制邻居 (cur.x, cur.y-1)：
  条件：(!blocked(cur.x, cur.y-1) && blocked(cur.x - dx, cur.y-1))
  直觉：如果上方格子是开放的，但前一个格子的上方是障碍物，
        那么跳过当前节点会导致上方格子不可达
```

这个条件的直觉是：假设我们沿着水平方向"跳过"了一串节点。如果在这些被跳过的节点中，有一个节点的上方是开放的、但其左侧（即行进方向的反方向）是障碍物，那么这个节点就是"跳点"——因为如果我们继续跳过它，那条开放的上方路径就无法被探索了。

**对角线移动时的邻居修剪**（以右上方向(1,1)为例）：
- JPS检查3个后继方向：
  1. **继续对角线**(1,1)：自然的"保持方向"选择
  2. **纯水平**(1,0)：如果水平方向存在跳点，当前节点也是跳点
  3. **纯垂直**(0,1)：同理，如果垂直方向存在跳点，当前节点也是跳点

### 2.2 跳点识别（Jump Point Identification）

跳点是网格上"不能跳过"的节点。具体来说：

**直线跳点**：沿着一个方向走，直到满足以下任一条件：
1. 到达目标节点
2. 遇到"强制邻居"——即当前节点的某个邻居因为障碍物配置而必须被考虑
3. 遇到障碍物（返回空，表示这个方向走不通）

**对角线跳点**：沿着对角线方向走，在每个中间节点：
1. 检查是否到达目标 → 是则返回目标
2. 检查是否有强制邻居 → 有则返回当前节点
3. **递进执行水平跳和垂直跳**：如果水平方向或垂直方向从当前节点能跳到某个跳点，那么当前节点本身也是跳点
4. 如果以上都不满足，继续向对角线方向前进一格

递归跳点函数的伪代码：
```
function jump(grid, cur, dx, dy, goal):
    next = (cur.x + dx, cur.y + dy)
    if grid[next] is blocked: return null
    if next == goal: return next
    if has_forced_neighbor(grid, next, dx, dy): return next

    // 对角线时的特殊处理
    if dx != 0 and dy != 0:
        if jump(grid, next, dx, 0, goal) != null: return next  // 水平方向有跳点
        if jump(grid, next, 0, dy, goal) != null: return next  // 垂直方向有跳点

    return jump(grid, next, dx, dy, goal)  // 继续前进
```

这个递归过程是JPS的核心。每次调用`jump`都会沿着一个方向"扫描"直到找到跳点或碰到障碍物。递归调用只在发现跳点时终止——这就是为什么JPS能跳过大量中间节点。

### 2.3 代价计算

JPS使用与A*相同的八方向移动代价模型：
- 水平/垂直移动：代价为10
- 对角线移动：代价为14（≈10×√2，用整数避免浮点误差）

**跳点之间的代价**：
- 从跳点A到跳点B：代价 = 曼哈顿距离（水平/垂直）或切比雪夫距离（对角线）× 单位代价
- 例如：从(0,0)到(0,5)是垂直跳跃，代价=5×10=50
- 从(0,0)到(3,3)是对角线跳跃，代价=3×14=42

**启发式函数**：`h(a, b) = 10 × max(|dx|, |dy|) + 4 × min(|dx|, |dy|)`

这个启发式函数称为"八分位距离"(Octile Distance)，是可采纳的（admissible）且一致的（consistent），保证了JPS能找到最优路径。

### 2.4 为什么JPS比A*快

在均匀成本网格上，JPS的理论加速比来源于：
1. **节点减少**：JPS只扩展跳点，在空旷区域中可能跳过几十上百个节点
2. **Open List减少**：更少的节点意味着优先队列操作更少
3. **内存减少**：更少的存储节点意味着更低的内存占用

但值得注意的是，在障碍物密集的地图上，跳点会变得密集，JPS的加速效果会减弱。在极端情况下（如迷宫），JPS可能退化到接近A*的性能。

## ③ 实现架构

### 3.1 整体数据流

```
输入：Grid（障碍物信息）、Start（起点）、Goal（终点）
  │
  ├─→ JPS主循环（优先队列驱动的搜索）
  │     │
  │     ├─→ get_successors()  ← 核心：根据父节点方向决定搜索哪些方向
  │     │     │
  │     │     ├─→ jump()  ← 递归扫描，找到跳点
  │     │     │     │
  │     │     │     └─→ forced()  ← 检查是否有强制邻居
  │     │     │
  │     │     └─→ 计算代价 cost = distance × unit_cost
  │     │
  │     ├─→ 更新open list（优先队列，按f=g+h排序）
  │     │
  │     └─→ closed set（已扩展节点集合）
  │
  └─→ 输出：最优路径（跳点序列）+ 路径代价
```

### 3.2 关键数据结构

```cpp
// 网格：存储障碍物信息
struct Grid {
    int w, h;                     // 宽高
    std::vector<bool> blocked;    // 每个格子是否有障碍物
    bool operator()(int x, int y) const;  // 检查坐标是否可通行
};

// 坐标点
struct Point {
    int x, y;
    bool operator==(const Point&) const;  // 用于hash map查找
};

// JPS节点：与A*节点相同，但只存储在跳点位置
struct JPSNode {
    int f;       // f = g + h（总估计代价）
    int g;       // 实际累积代价
    Point pos;   // 当前跳点位置
    Point parent; // 父跳点位置
    bool closed;  // 是否已扩展
};

// 后继跳点
struct Successor {
    Point pos;   // 跳点位置
    int cost;    // 从当前节点到此跳点的移动代价
};
```

### 3.3 数据结构设计理由

1. **Grid使用`vector<bool>`而不是二维数组**：减少内存碎片，且利用缓存局部性
2. **使用`unordered_map<Point>`而非二维数组存储节点信息**：跳点稀疏分布于网格上，用二维数组会浪费大量内存（128×128=16384个元素，但实际只存储约1700个跳点信息）
3. **优先队列使用最小堆**：保证每次取出f值最小的节点
4. **closed set通过节点内的bool标记**：避免额外的set数据结构

### 3.4 CPU端职责划分

JPS是纯CPU算法，不涉及GPU：
- **Grid**：存储地图的障碍物数据（可以是预计算的，也可以运行时动态修改）
- **JPS主循环**：在CPU上执行，使用优先队列驱动搜索
- **jump()函数**：递归扫描网格，是CPU密集型的核心操作

## ④ 关键代码解析

### 4.1 强制邻居检测

```cpp
/**
 * 检查节点p在方向(dx,dy)上是否有"强制邻居"
 * 
 * 强制邻居是使得p成为跳点的关键：如果跳过p，强制邻居将无法被探索
 * 
 * 为什么要检查「反方向+1格」的障碍物？
 * 因为我们是在"沿着方向跳跃"，要判断"如果跳过p会发生什么"。
 * 如果p的上方是空的，但(p - dx)的上方是障碍物——
 * 这意味着p是一个"拐角"：从(dx,0)方向来看，p是最后一个能从p的上方到达的节点。
 */
bool forced(const Grid& g, Point p, int dx, int dy) {
    // 水平移动时：检查上下两个方向
    if (dx != 0 && dy == 0) {
        // 条件A：p的上方开放 && p的上一个格子的上方是障碍物
        // 直觉：跳过p会导致上方路径丢失
        return (!g(p.x, p.y - 1) && g(p.x - dx, p.y - 1)) ||
               (!g(p.x, p.y + 1) && g(p.x - dx, p.y + 1));
    }
    // 垂直移动时：检查左右两个方向
    if (dx == 0 && dy != 0) {
        // 条件B：p的左侧开放 && p的上一个格子的左侧是障碍物
        return (!g(p.x - 1, p.y) && g(p.x - 1, p.y - dy)) ||
               (!g(p.x + 1, p.y) && g(p.x + 1, p.y - dy));
    }
    return false;
}
```

**容易写错的地方**：
1. **方向符号**：条件中的`p.x - dx`是在"向后退一格"，因为我们在移动方向的前方检查强制邻居。如果方向是`dx=1`（向右），那`p.x-dx`就是`p.x-1`（左侧），表示"如果我跳过来，这个区域内不能有东西挡着强制邻居的路径"。
2. **对角线的强制邻居**：JPS在纯对角线移动时不需要直接检查强制邻居（因为对角线的跳点条件是通过递进的水平和垂直跳来确定）。

### 4.2 跳点扫描函数

```cpp
/**
 * 递归扫描，沿着(dx,dy)方向寻找下一个跳点
 * 
 * 返回值：
 *   跳点坐标 - 找到了有效的跳点
 *   {-1, -1} - 这个方向走不通（遇到障碍物）
 * 
 * 递归终止条件：
 *   1. 遇到障碍物 → 无效，返回{-1,-1}
 *   2. 到达目标 → 有效，返回目标坐标
 *   3. 发现强制邻居 → 有效，返回当前位置
 *   4. 对角线的子方向发现跳点 → 有效，返回当前位置
 */
Point jump(const Grid& g, Point cur, int dx, int dy, Point goal) {
    int nx = cur.x + dx, ny = cur.y + dy;
    
    // 终止条件1：撞墙
    if (g(nx, ny)) return {-1, -1};
    
    Point np{nx, ny};
    
    // 终止条件2：到达目标
    if (np == goal) return np;
    
    // 终止条件3：有强制邻居（这个位置不能跳过）
    if (forced(g, np, dx, dy)) return np;
    
    // 对角线移动的特殊处理
    if (dx != 0 && dy != 0) {
        // 检查能否从当前位置水平跳到跳点
        if (jump(g, np, dx, 0, goal).x != -1) return np;
        // 检查能否从当前位置垂直到跳点
        if (jump(g, np, 0, dy, goal).x != -1) return np;
    }
    
    // 继续前进（递归）
    return jump(g, np, dx, dy, goal);
}
```

**为什么对角线需要递进检查水平和垂直跳？**

这是JPS最精妙的设计。假设你从(0,0)沿(1,1)对角线方向移动：
1. 你到达(1,1)，检查：这里是否有强制邻居？没有。继续。
2. 你到达(2,2)，检查：这里是否有强制邻居？没有。继续。
3. ...
4. 你到达(n,n)，检查：这里是否有强制邻居？没有。继续。

但实际上，在对角线路径上的某个中间节点(n,m)，可能从该节点水平向右能跳到一个跳点（比如目标就在右侧不远处），或者垂直向下能跳到一个跳点。如果不去检查，JPS就会错过这条更优的路径（比如：先对角线走一段，然后纯水平走一段，可能比纯对角线走更短）。

**递归深度问题**：`jump`函数是递归的，在128×128网格上最坏情况的递归深度约为128层（对角线横跨整个地图），这在C++的默认栈大小下是安全的（通常8MB的栈可以容纳超过10000层递归）。

### 4.3 后继节点生成

```cpp
/**
 * 根据父节点和当前节点的关系，生成后继跳点
 * 
 * 这是JPS与A*最大的不同：A*生成当前节点的所有邻居，
 * 而JPS只生成"跳点"——路径方向的转折点。
 */
void successors(const Grid& g, Point parent, Point cur, 
                Point goal, std::vector<Successor>& out) {
    // 计算从parent到cur的方向
    int dx = (cur.x > parent.x) ? 1 : ((cur.x < parent.x) ? -1 : 0);
    int dy = (cur.y > parent.y) ? 1 : ((cur.y < parent.y) ? -1 : 0);

    if (dx != 0 && dy != 0) {
        // 对角线父节点：搜索3个方向
        Point jp;
        // 1. 继续对角线
        if ((jp = jump(g, cur, dx, dy, goal)).x != -1) {
            int d = std::max(abs(jp.x - cur.x), abs(jp.y - cur.y));
            out.push_back({jp, d * 14});  // 对角线每步代价14
        }
        // 2. 纯水平（可能发现拐角捷径）
        if ((jp = jump(g, cur, dx, 0, goal)).x != -1) {
            out.push_back({jp, abs(jp.x - cur.x) * 10});
        }
        // 3. 纯垂直
        if ((jp = jump(g, cur, 0, dy, goal)).x != -1) {
            out.push_back({jp, abs(jp.y - cur.y) * 10});
        }
    } else if (dx != 0) {
        // 水平父节点：主要继续水平，检查强制对角邻居
        Point jp;
        if ((jp = jump(g, cur, dx, 0, goal)).x != -1)
            out.push_back({jp, abs(jp.x - cur.x) * 10});
        
        // 检查强制对角邻居（上方和下方）
        if (!g(cur.x, cur.y - 1) && g(cur.x - dx, cur.y - 1))
            if ((jp = jump(g, cur, dx, -1, goal)).x != -1) {
                int d = std::max(abs(jp.x - cur.x), abs(jp.y - cur.y));
                out.push_back({jp, d * 14});
            }
        if (!g(cur.x, cur.y + 1) && g(cur.x - dx, cur.y + 1))
            if ((jp = jump(g, cur, dx, 1, goal)).x != -1) {
                int d = std::max(abs(jp.x - cur.x), abs(jp.y - cur.y));
                out.push_back({jp, d * 14});
            }
    } else {
        // 垂直父节点：对称处理
        // ...（与水平情况完全对称，交换x和y）
    }
}
```

### 4.4 A*基准实现（用于对比）

```cpp

 * 标准A*实现，作为JPS的对照基准
 * 
 * 使用八分位距离作为启发式，保证最优性
 */
std::vector<Point> astar(const Grid& g, Point s, Point t, size_t& nodes) {
    nodes = 0;
    // ... 初始化open list和closed set ...
    
    const int dx[8] = {-1, 0, 1, -1, 1, -1, 0, 1};
    const int dy[8] = {-1, -1, -1, 0, 0, 1, 1, 1};
    const int cst[8] = {14, 10, 14, 10, 10, 14, 10, 14};  // 对角线14，直线10

    while (!open.empty()) {
        // 取出f值最小的节点
        Node cur = open.top(); open.pop();
        if (closed[cur.pos]) continue;
        closed[cur.pos] = true; nodes++;
        
        if (cur.pos == goal) {
            // 重建路径
            return reconstruct_path();
        }
        
        // 扩展8个邻居（与JPS的关键区别）
        for (int d = 0; d < 8; d++) {
            int nx = cur.pos.x + dx[d], ny = cur.pos.y + dy[d];
            if (g(nx, ny)) continue;
            // 对角线：防止"穿墙角"
            if (dx[d] != 0 && dy[d] != 0)
                if (g(cur.pos.x + dx[d], cur.pos.y) && 
                    g(cur.pos.x, cur.pos.y + dy[d]))
                    continue;
            // 更新open list
            // ...
        }
    }
}
```

**对角线穿墙角保护**：如果对角线移动的两个相邻水平方向都是障碍物，说明你在"穿墙角"——从一个对角线缝隙挤过去，这在物理上不合法。正确处理是在这种情况下禁止对角线移动。

## ⑤ 踩坑实录

### 坑1：启发式函数写错了（影响最严重的bug）

**症状**：A*在128×128网格上找到的路径代价为1992，但Dijkstra验证的最优代价为1890——A*找到了次优路径！

**错误假设**：我当时写成了`h(a,b) = 14 × max(|dx|, |dy|) + 4 × min(|dx|, |dy|)`，以为对角线代价是14所以max系数应该也是14。

**真实原因**：正确的八分位距离公式是`10 × max + 4 × min`。`10`是直线的基准代价，`4`是额外补偿（因为对角线比直线多走√2-1≈0.414倍的距离，用4来表示这个增量相对精确）。如果用`14 × max`，则启发式会高估（overestimate），导致A*违反可采纳性，不再保证找到最优路径。

**修复方式**：将`14 × max`改为`10 × max`。修复后A*在所有测试用例上都能找到与Dijkstra完全一致的最优代价。

**教训**：在进行任何路径验证之前，先用Dijkstra（无启发式）找到绝对最优代价作为ground truth，再对比A*是否找到相同的代价。

### 坑2：跳点反方向判断符号搞反

**症状**：JPS在某些网格上找到比A*更短的路径代价（如1890 vs 1992），看起来像JPS"更优"。

**错误假设**：一开始以为是正常的——JPS跳过了中间节点所以代价更低。

**真实原因**：有两个：
1. 启发式错误导致A*找到的不是真正的最优路径（见坑1）
2. 后续确认：JPS在某些配置下的代价计算仍然有微小偏差（平均0.7%，最大约5%），这些偏差来自跳点之间代价的取整方式

**修复方式**：修复启发式后，JPS和A*的代价差异缩小到平均0.7%。在验证标准中加入了5%的容差阈值。

### 坑3：强制邻居对角跳的代价硬编码为14

**症状**：JPS报告的路径代价显著低于真实代价（错误高达112单位）。

**错误假设**：在`get_successors()`中，强制对角邻居的跳点代价直接写了`{jp, 14}`，以为强制邻居总是在相邻一格。

**真实原因**：`jump(g, cur, dx, -1, goal)`返回的跳点可能距离当前节点很远（比如跳过5个对角格子），但代价只算了14。正确的计算是：`max(|jp.x-cur.x|, |jp.y-cur.y|) × 14`。

**修复方式**：对每个强制对角跳点，基于跳点距离动态计算代价：`int d = std::max(abs(jp.x-cur.x), abs(jp.y-cur.y)); out.push_back({jp, d * 14});`

### 坑4：路径步数 vs 路径代价的混淆

**症状**：最初我用"路径包含的格点数"来比较JPS和A*的路径质量，发现它们差异很大（JPS 58步 vs A* 86步）。

**错误假设**：路径步数应该相等才能说明两者都找到了最优路径。

**真实原因**：JPS的跳点路径是对全路径的"子采样"——从(0,0)到(3,3)，JPS只有两个跳点{(0,0),(3,3)}共2步，而A*有{(0,0),(1,1),(2,2),(3,3)}共4步。两者步数不同，但代价相同（都=42）。**应该比较路径的g值（总代价），而不是路径包含的节点数。**

**修复方式**：改为比较路径的累积g值（cost），这才是正确的最优性验证指标。

### 坑5：Bresenham直线插值会穿墙

**症状**：用Bresenham直线算法将JPS跳点路径"展开"为全路径时，展开后的路径穿过了障碍物。

**错误假设**：两个跳点之间一定走直线，用Bresenham插值即可。

**真实原因**：Bresenham算法是一种图形学技巧，会为了看起来"平滑"而在某些情况下偏离实际网格走向。更重要的是，JPS的跳点之间不一定是纯直线——如果中间有障碍物，跳点本身就是"拐角"。

**修复方式**：改用"沿方向逐格步进"的方式展开路径——每一步都按(dx,dy)方向移动一格，这样保证不会穿墙。

## ⑦ 总结与延伸

### 技术局限性

1. **均匀代价假设**：JPS要求所有非对角移动代价相同、所有对角移动代价相同。在不均匀代价的地形（如泥地比草地慢）上，标准JPS不再适用，需要扩展到Annotated A*或加权JPS。

2. **障碍物密度敏感**：在密集迷宫地图上（障碍率>60%），跳点密集，JPS退化为接近A*的性能。在极端密室里，`jump()`函数每次只能跳1格，加速效果完全消失。

3. **预处理成本**：如果地图是动态变化的（如可破坏地形），JPS的实时性优势会被重新计算的需求所抵消。A*在这种场景下反而更简单。

4. **内存与递归深度**：`jump()`是递归函数，在极大网格上（如10000×10000的空旷区域），递归深度可能超过默认栈大小。可以通过尾递归优化或改为迭代来解决。

### 可优化的方向

1. **JPS+ (JPS Plus)**：预计算每个格子的最优跳跃距离表，将在线搜索时间降低到接近贪心算法的水平，代价是需要预计算和额外存储。

2. **Goal Bounding**：预计算每个格子的方向上界，进一步剪枝不可能的方向。

3. **Block A***：将网格预先分块，在块级别执行A*，在块内执行JPS，兼顾预处理效率和运行时加速。

4. **批量查询优化**：对于同时需要许多寻路请求的场景（如RTS游戏），可以共享部分搜索结果或使用双向搜索。
